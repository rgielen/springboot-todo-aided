# HTMX Frontend for Todo Management

## Prompt

Ergänze ein HTMX-basiertes Frontend für das TODO Management. Orientiere dich an der UX von TodoMVC. Sorge für ein schickes, cooles und ansprechendes WebDesign und perfekte Usability. Benutze playwright-cli für die Überprüfung und Perfektionierung. Erstelle umfangreiche e2e Tests zur Absicherung der Funktionalität.

## Context

Das Projekt ist aktuell eine reine REST-API ohne UI. Ziel: HTMX-basiertes Frontend mit TodoMVC-UX, ansprechendem Design und umfassenden Playwright-E2E-Tests. Die bestehende REST-API (`/api/todos`) bleibt unverändert.

## 2. Repository-Erweiterungen (`TodoRepository.java`)

Neue `@Query`-Methoden (kein Query Derivation):

- `findAllOrderByCreatedAtAsc()` — alle Todos, sortiert nach Erstellzeitpunkt
- `findAllByCompleted(boolean)` — gefiltert, ohne Pagination
- `countActive()` — Anzahl nicht-erledigter Todos
- `updateAllCompleted(boolean)` — Bulk-Toggle (`@Modifying`)
- `deleteAllCompleted()` — erledigte löschen (`@Modifying`)
- `areAllCompleted()` — Prüfung ob alle erledigt

## 3. Neuer Controller: `TodoViewController.java`

`@Controller` (nicht `@RestController`), separater Controller neben bestehendem `TodoController`.

### Endpoints

| Method | Pfad | Zweck | Antwort |
|--------|------|-------|---------|
| GET | `/` | Volle Seite | `index` Template |
| POST | `/todos` | Todo anlegen | Fragments: todo-list, footer, input-reset |
| PUT | `/todos/{id}/toggle` | Toggle einzeln | Fragments: todo-list, footer |
| PUT | `/todos/{id}` | Inline-Edit | Fragment: todo-item |
| DELETE | `/todos/{id}` | Löschen | Fragment: footer (Item per hx-swap="delete") |
| POST | `/todos/toggle-all` | Alle togglen | Fragments: todo-list, footer |
| POST | `/todos/clear-completed` | Erledigte löschen | Fragments: todo-list, footer |
| GET | `/todos` | Filter wechseln | Fragments: todo-list, footer |

### Architektur-Entscheidungen

- `FragmentsRendering` für Multi-Fragment-Responses (OOB Swaps)
- Filter-State client-seitig via `htmx:configRequest` Event (kein Server-Session-State)
- Form-encoded Parameters (`@RequestParam`), kein JSON
- `@Transactional` auf schreibenden Methoden
- Kein Service-Layer (analog zu bestehendem `TodoController`)

## 4. Templates

### Dateistruktur

```
src/main/resources/templates/
├── index.html                    # Hauptseite (composit)
└── fragments/
    ├── header.html               # Input-Feld + Toggle-All
    ├── todo-list.html            # Liste mit Iteration
    ├── todo-item.html            # Einzelnes Todo-Item
    └── footer.html               # Count, Filter, Clear Completed
```

### TodoMVC Features

- **Neues Todo**: Input im Header, Enter-Taste, `hx-post="/todos"`
- **Toggle**: Checkbox pro Item, `hx-put="/todos/{id}/toggle"`
- **Toggle All**: Checkbox im Header, `hx-post="/todos/toggle-all"`
- **Inline Edit**: Doppelklick auf Label → Edit-Modus (JS), Enter speichert (`hx-put`), Escape bricht ab
- **Löschen**: Destroy-Button, `hx-delete`, `hx-swap="delete"` auf `closest li`
- **Filter**: All / Active / Completed Links im Footer, `hx-get="/todos?filter=..."`
- **Clear Completed**: Button im Footer, `hx-post="/todos/clear-completed"`
- **Item Count**: "X items left" im Footer, aktualisiert via OOB-Swap
- **Footer/Main ausgeblendet**: Wenn keine Todos vorhanden

## 5. Static Resources

### CSS: `src/main/resources/static/css/todo.css`

- Modernes, TodoMVC-inspiriertes Design (kein Framework)
- System-Font-Stack, Card-Layout mit max-width 550px, Schatten, abgerundete Ecken
- Custom Checkboxes (CSS-only), Hover-Effekte, Transitions
- Responsive, `.htmx-request` Loading-Indicator
- Completed-Items: Durchgestrichen, gedämpfte Farbe

### JS: `src/main/resources/static/js/todo.js`

Minimal (~40 Zeilen), nur für:
- Doppelklick → Edit-Modus (Class-Toggle + Focus)
- Escape → Edit abbrechen
- Filter-Parameter automatisch an alle HTMX-Requests anhängen (`htmx:configRequest`)

## 6. Playwright E2E Tests

### Test-Setup

- `TodoViewBaseTest` (abstrakt): `@SpringBootTest(RANDOM_PORT)`, `@Import(TestcontainersConfiguration)`, Playwright Browser/Page Lifecycle, DB-Cleanup
- Gleiche Testcontainers-Infrastruktur wie bestehende API-Tests

### `TodoViewIntegrationTest` — Testfälle

1. **addTodo_appearsInList** — Todo anlegen, in Liste sichtbar
2. **addMultipleTodos_allAppearInList** — 3 Todos, Count korrekt
3. **toggleTodo_marksAsCompleted** — Checkbox, Class "completed", Count
4. **toggleTodo_marksAsActive** — Toggle hin und zurück
5. **toggleAllTodos_marksAllCompleted** — Alle auf erledigt
6. **toggleAllTodos_marksAllActive** — Alle zurück auf aktiv
7. **inlineEdit_doubleClickAndEnter** — Doppelklick, Bearbeiten, Enter
8. **inlineEdit_escapeReverts** — Escape bricht ab, Originalwert bleibt
9. **deleteTodo_removesFromList** — Löschen-Button, Item weg
10. **filterActive_showsOnlyActive** — Active-Filter
11. **filterCompleted_showsOnlyCompleted** — Completed-Filter
12. **filterAll_showsAllTodos** — All-Filter
13. **clearCompleted_removesCompletedTodos** — Clear-Button
14. **footerHiddenWhenEmpty** — Footer nicht sichtbar bei 0 Todos
15. **itemCount_singularAndPlural** — "1 item left" vs "2 items left"
16. **persistence_survivesPageReload** — Seite neu laden, Todos bleiben
